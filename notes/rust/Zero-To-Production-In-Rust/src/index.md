# 读书笔记

[Rust 从零到生产 - 后端开发简介](https://www.zero2prod.com)

[书籍源码下载](https://github.com/LukeMathWalker/zero-to-production)

该书通篇从实践中讲解如何从头开始构建一个功能齐全的电子邮件通讯`API`。

##  学习方法

基于问题开展学习：选择一个想解决的问题，让问题驱动新概念和技术的引入。

## 需求分析

书中以*用户故事*的形式确定需求。格式相当简单：

> As a 作为 ……，
> I want to 我想 ……，
> So that 以便 ……

用户故事可以帮助我们捕捉为谁构建（*作为*）、想要执行的操作（*我想*）以及动机（*以便*）。

## 业务需求

本书将完成三个用户故事：

- 作为博客访问者，
  我想订阅时事通讯，
  以便在博客上发布新内容时收到电子邮件更新；
- 作为博客作者，
  我想向所有订阅者发送电子邮件，
  以便在发布新内容时通知他们；
- 作为订阅者，
  我希望能够取消订阅时事通讯，
  这样我就可以停止接收来自博客的电子邮件更新。

不会添加功能

- 管理多个时事通讯；
- 在多个受众中细分订户；
- 跟踪打开率和点击率。

## 非功能性需求

我们期望：

- 在容易出错的环境中运行时实现高可用性；
- 让我们能够持续发布零停机时间的新版本；
- 处理动态工作负载（例如请求量）。

**高可用性**意味着我们的应用程序应该能够在不停机的情况下处理请求，即使一台或多台机器突然开始出现故障（在云环境中很常见）。这迫使我们的应用程序是*分布式*的——可在多台机器上运行它的多个实例。

如果我们希望能够处理动态工作负载，情况也是如此——我们应该能够**衡量**系统是否处于负载状态，并通过启动应用程序的新实例来解决问题。这也要求我们的基础设施具有弹性，以避免过度配置及其相关成本。

运行复制的应用程序会影响我们的数据持久性方法——我们将避免使用本地文件系统作为主要存储解决方案，而是依赖数据库来满足持久性需求。

## 开发流程

让我们放大其中一个用户故事：

> 作为博客作者，
> 我想向所有订阅者发送电子邮件，
> 以便在发布新内容时通知他们。

这*在实践*中意味着什么？我们需要建造什么？
一旦开始仔细查看，就会出现大量问题 - 例如，我们如何确保调用者确实是博客作者？我们需要引入认证机制吗？我们是支持电子邮件中的 HTML 还是我们坚持纯文本？表情符号呢？

我们不会深入研究一个故事，而是尝试构建足够的功能来*在一定程度上*满足我们第一个版本中所有故事的要求。
然后我们将返回并改进：添加容错和重试电子邮件传递，为新订阅者添加确认电子邮件等。

**我们将在迭代中开发**：每次迭代都需要固定的时间，并为我们提供稍微更好的产品版本，从而改善用户的体验。

值得强调的是，我们迭代的是产品特性，而不是工程质量：每次迭代中生成的代码都将经过测试并正确记录，即使它只提供了一个微小的、功能齐全的特性。
我们的代码将在每次迭代结束时投入生产——它需要具有产品质量。
